# STM32 智能小车 — 红外循迹 + 超声波避障

> 基于 STM32F103C8T6 的四路红外循迹 + 超声波避障智能小车，支持蓝牙遥控、OLED 显示、舵机云台。

## 硬件清单

| 模块 | 型号 | 数量 |
|------|------|------|
| 主控 | STM32F103C8T6 (Cortex-M3) | 1 |
| 电机驱动 | TB6612FNG | 1 |
| 直流电机 | TT 马达 1:48 | 4 |
| 红外传感器 | TCRT5000 × 4 路 | 1 |
| 超声波 | HC-SR04 | 1 |
| 舵机 | SG90 (云台) | 1 |
| OLED | 0.96寸 I2C (SSD1306) | 1 |
| 蓝牙 | JDY-31 (BLE) | 1 |

## 模式与蓝牙指令

| 指令 | 模式 | 说明 |
|------|------|------|
| `1` | 自动模式 | 无障碍 → 循迹，有障碍 → 避障（默认） |
| `2` | 纯循迹 | 四路红外跟踪黑线 |
| `3` | 纯避障 | 超声波测距 + 舵机扫描绕障 |
| `4` | 蓝牙遥控 | F=前 B=后 L=左 R=右 S=停 |

`1~4` 在任何模式下都生效。`F/B/L/R/S` 仅在蓝牙模式(4)下生效。

## 循迹算法

4 路红外传感器分级响应，9 种情况按优先级匹配：

```
[1 0 0 1] 完美居中 → 直走
[1 0 1 1] X2踩线  → 左微调 (差速)
[1 1 0 1] X3踩线  → 右微调 (差速)
[0 0 1 1] 左偏    → 急左转 (坦克式)
[1 1 0 0] 右偏    → 急右转 (坦克式)
[0 1 1 1] 极左偏  → 半速左转修正
[1 1 1 0] 极右偏  → 半速右转修正
3路以上踩线    → 十字路口，直走
[1 1 1 1] 离线    → 原地找线（超时2秒直走）
```

## 避障算法

1. 超声波测距 → 前方 < 25cm 触发
2. 舵机左扫(0°) → 右扫(180°) → 比较两侧距离
3. 选择空旷方向 (> 35cm) → 旋转 → 前进绕行
4. 两侧均堵 → 180° 掉头

## 开发中遇到的问题与解决

### 1. 蓝牙指令偶尔收不到

**现象**：手机蓝牙发指令后小车没反应，多发几次才偶尔响应一次。

**排查**：用串口调试助手直连 STM32 TX 引脚，发现数据确实进了中断、`RxData` 也更新了，但主循环的 `if` 判断偶尔跳过去。

**根因**：`RxData` 没加 `volatile`。编译器优化后把 `RxData` 的值缓存到了寄存器里，主循环一直用的是寄存器里的旧值，中断更新的新值被"优化掉了"。

**解决**：`u8 RxData` → `volatile u8 RxData`，同时在头文件里声明为 `extern volatile u8 RxData`。编译后蓝牙响应恢复正常。

**延伸**：共享于中断和主循环之间的变量必须加 `volatile`——这是 Cortex-M 裸机编程的基本功，面试必问。

### 2. Flash Download Failed — No Algorithm found

**现象**：编译 0 Error 0 Warning，一点下载就报 `No Algorithm found for: 08000000H`。

**排查**：检查 `C:\Keil_v5\ARM\Flash\` 目录，发现根本没有 `STM32F10x_*.FLM` 文件。Pack 已经下载到 `.Download` 目录并且解压到了 `PACK\Keil\STM32F1xx_DFP\2.2.0\Flash\` 下，但 Keil 不会自动把 FLM 文件复制到它的搜索路径。

**解决**：将 Pack 目录下的 6 个 `.FLM` 文件复制到 `C:\Keil_v5\ARM\Flash\`，然后在 Keil → Debug → Settings → Flash Download → Add → 选择 `STM32F10x_128.FLM`。

**踩坑**：直接改 `.uvprojx` 工程文件里的 `FlashDriverDll` 行没有用——Keil 每次打开工程都会根据器件数据库重新生成这一行，手动改的内容会被覆盖。

### 3. 舵机选型 — 为什么 PWM 频率是 50Hz

**现象**：舵机初始化后转到 90° 偶尔到位，偶尔抖动甚至不动。

**排查**：SG90 数据手册里写 PWM 周期必须是 20ms（50Hz），高电平 0.5ms~2.5ms 对应 0°~180°。我在 `Servo_Init(1999, 719)` 中确认了计算：72MHz ÷ (719+1) = 100KHz 计数器时钟，100KHz ÷ (1999+1) = 50Hz —— 频率没错。

**解决**：问题是主循环里 `Delay_ms()` 阻塞了 PWM 更新，改成在避障函数中先停车再转舵机，确保舵机到位后才执行后续逻辑。

**经验**：器件时序不能只算一次就完了——还要看代码里有没有阻塞操作影响了信号连续性。

### 4. 电机 PWM 频率的选择

**问题**：PWM 频率选多少合适？太低有啸叫，太高驱动芯片烫手。

**分析**：人耳听觉范围约 20Hz~20KHz，2KHz 以上基本不敏感。TB6612 驱动芯片的开关损耗随频率升高而增大。折中选 5KHz：人耳听不到（不扰人），驱动芯片发热在可接受范围内。

### 5. 离线死循环

**现象**：小车脱离黑线跑道后原地打转，永远不停，只能断电。

**代码**：`while(TRACE_X2 == 1 && TRACE_X3 == 1)` — 中间两路传感器找不到黑线就死循环旋转。

**解决**：加超时计数器 `timeout < 2000`（约 2 秒，足够转 360°），超时就退出循环直走。这样就算没有跑道，车也不会卡死在一个地方。

### 6. 避障与循迹的优先级

**问题**：车在跑道上遇到前方障碍物，应该继续循迹还是先躲？

**决策**：避障优先。`Control_AutoMode()` 里先判断 `obstacleDist < 25cm`，有障碍直接进 `Car_AvoidObstacle()`，没障碍才走 `Control_TraceMode()`。逻辑很简单——前面有墙，无论黑线在哪，安全第一。

### 7. 超声波测距零值处理

**现象**：小车在空旷场地上 OLED 显示距离 0cm，然后避障逻辑被误触发。

**根因**：HC-SR04 超量程（>4 米无回波）时返回 0。避障判断 `obstacleDist < 25` 对 0 值也成立了（0 < 25 = true）。

**解决**：判断条件改成 `obstacleDist > 0 && obstacleDist < 25`，多了 `> 0` 把超量程情况排除。`obstacleDist == 0` 的含义是"前方空旷"而非"前方贴着墙"。

### 8. 四路红外的局限性

**现象**：急弯时车频繁丢线，四个传感器全白，触发离线旋转。

**原因**：4 路红外只提供 4 级离散反馈（0/1），无法反映"偏了多少"，本质是一个粗粒度的开关量系统。6 路也只是把分辨率从 4 格变 6 格，治标不治本。

**改进方向**：加 K210 或 OpenMV 摄像头做视觉巡线，输出黑线中心在画面中的连续像素坐标，上 PID 做闭环控制，才能真正解决急弯和高速情况下的跟踪问题。

### 9. 转弯靠延时估算

**问题**：`Car_AvoidObstacle()` 里转弯用 `Delay_ms(400)` 估算 90°，没有编码器或陀螺仪反馈。

**影响**：电池电压波动 → 电机转速变化 → 同样的 400ms 延时实际转的角度不一样。

**改进方向**：加 MPU6050 陀螺仪做实时航向角反馈，转到目标角度就停，不依赖延时估算。

### 10. Keil 编译器版本兼容

**问题**：工程用的是 ARM Compiler 5 (V5.06)，这是 Keil MDK 5.24a 默认的。但新版 MDK (5.36+) 已经默认 ARM Compiler 6，编译标准外设库会报一堆错误。

**解决**：工程设置里 `Options → Target → Code Generation → ARM Compiler` 选 `V5.06`，或者用 MDK 5.24a 这个特定版本。标准外设库已经停止维护，官方不提供 Compiler 6 的兼容支持。

### STM32 基础

**Q: STM32F103C8T6 的系统时钟是多少？TIM 时钟怎么算？**

外接 8MHz 晶振 → PLL 倍频到 72MHz（`SystemInit()` 默认）。TIM1/2/3/4 挂 APB1（最大 36MHz），但因为 APB1 预分频器 ≠ 1，定时器时钟自动 ×2 = 72MHz。所以定时器的时基还是 72MHz。

**Q: I2C 怎么驱动 OLED 的？为什么用软件 I2C 而不是硬件？**

用 GPIO 模拟 I2C 时序（软件 I2C）：拉低 SDA → 拉低 SCL → 发 8 位数据 → 拉高 SCL 锁存。不用硬件 I2C 的原因：F103 的硬件 I2C 有历史 bug（在特定时序下会卡死），软件 I2C 更稳定且移植性好。

### 系统设计

**Q: 如果让你重做这个项目，你会怎么改进？**

1. 加 K210/OpenMV 摄像头做视觉循迹，输出连续偏移量 → 上 PID
2. 6 路红外替代 4 路，提高离散分辨率
3. 加 MPU6050 陀螺仪做航向角反馈，转弯不再靠延时估算
4. FreeRTOS 多任务替代裸机 while 循环
5. 串口中断加 ORE/溢出处理，通信更健壮
